# Task Cockpit · 设计文档

**日期**：2026-06-11
**形态**：Claude Code Skill（自带常驻本地服务 + 看板模板），将来可套 plugin 分发
**一句话**：你在 Claude Code 里把脑子里一堆活倒给 agent，它拆成任务让你审、入库，并维护一个只读看板帮你回答"该先干啥"；每完成一个任务即时沉淀一条可用的成就陈述，越攒越厚，随时能拼成述职/周报/复盘材料。

---

## 1. 背景与目标

互联网打工人常同时对接多个项目，每个项目多个任务。"哪个做了、还剩什么、接下来干啥、哪个急"这些项目管理的负担，持续分散本该用于创造的精力。

本产品的目标是把这部分负担转交给 agent：
- 用户只需用自然语言"倒事"，agent 负责理解、拆解、排序。
- 一个只读看板随时呈现全局，并主动算出"今日该聚焦什么"。
- 完成的工作即时沉淀为成就记录，解决"述职/答辩时想不起自己做过什么"的痛点，长期看也是个人与团队的经验财富。

**非目标（当前范围之外）**：
- 团队共享同一份数据（需共享后端，推翻纯本地设计）。
- 看板上的直接交互（点选/拖拽编辑）——当前看板只读，所有修改经对话完成。
- 与飞书等外部系统的数据同步。

---

## 2. 总体架构

对话驱动 + 只读看板，二者通过本地数据文件解耦：

```
你 💬 Claude Code  ──→  Agent（按 SKILL.md 工作流）  ──→  数据文件（~/.task-cockpit/）
                                                              │
                                          本地常驻服务 server.py 托管
                                                              │
                                                      🖥️ 浏览器看板（每 2s 轮询自动刷新）
```

- **对话是唯一输入口**：拆任务、改任务、标完成、生成 CV 都在 CC 对话里发起。
- **看板是只读产物**：呈现数据，不接受编辑。
- **数据是核心资产**：与 skill 代码分离，升级 skill 不影响数据。

---

## 3. 数据层

数据目录固定在用户家目录下：`~/.task-cockpit/`（**禁止写死个人绝对路径**，一律由当前用户家目录推算，以保证将来分发给他人时无需返工）。

### 3.1 `projects.json` — 项目清单
```json
{
  "proj_app": { "name": "App 改版", "createdAt": "2026-06-01", "archived": false },
  "proj_report": { "name": "年度报告", "createdAt": "2026-06-09", "archived": false }
}
```

### 3.2 `tasks.json` — 当前任务快照（可改可删）
```json
{
  "task_a1": {
    "project": "proj_app",
    "title": "登录页改稿",
    "status": "进行中",
    "priority": "高",
    "due": "2026-06-12",
    "nextAction": "等设计给图",
    "blocked": true,
    "draft": false,
    "createdAt": "2026-06-10"
  }
}
```

字段取值约定：
- `status`：`未开始` / `进行中`（注：任务一旦完成即移出 `tasks.json` 并写入 `achievements.jsonl`，故 `tasks.json` 中常态只有前两个值；`已完成` 仅作为完成动作发生的瞬时态，不长期存于此文件）
- `priority`：`高` / `中` / `低`
- `due`：ISO 日期字符串，可为空
- `nextAction`：一句话，下一步具体动作，可为空
- `blocked`：布尔，是否在等别人/别的事
- `draft`：布尔，`true` 表示 agent 刚拆出、尚待用户确认（看板高亮）

### 3.3 `achievements.jsonl` — 成就库（只增不减，每行一条）
任务完成时追加。既是完成历史，也是 CV 的原料。
```jsonl
{"id":"done_x1","date":"2026-06-09","project":"App 改版","title":"登录页改稿","outcome":"完成新设计走查并提测，统一全表单报错文案","reflection":"设计稿和实现有出入，下次早点拉设计对一遍","cv":"主导登录页改版落地，统一全表单报错文案，降低用户填写出错率","cvStatus":"ready"}
```
字段：
- `outcome`：客观成果（对外，CV 用）
- `reflection`：主观复盘/教训（对内，可选，成长材料与团队沉淀用）
- `cv`：agent 生成的可直接使用的成就陈述
- `cvStatus`：`ready`（已沉淀）/ `pending`（素材不足、挂起待补）

### 3.4 `cv-exports/` — 可选导出存档
生成的述职/周报/复盘 markdown 可存档于此，便于回找。

**核心设计原则**：当前快照（`tasks.json`）与历史沉淀（`achievements.jsonl`）严格分离。删改任务不影响成就历史——这是"完成即沉淀、长期可回溯"得以成立的根本。

## 4. 交互流程（三条主线，均在 CC 对话发起）

### 4.1 倒事 → 拆解 → 确认
1. 用户用自然语言描述一堆待办（含哪些项目、各有什么事、截止时间等）。
2. agent 理解后拆成"归属项目 + 若干任务"，以 `draft: true` 写入 `tasks.json`，看板高亮显示为"草稿待确认"。
3. agent 在终端复述拆解结果，用户在对话里调整（改优先级、加截止日、拆更细、删条目等），agent 改完刷新看板。
4. 用户说"确认"，agent 将相关任务 `draft` 翻为 `false`，看板高亮消失。

### 4.2 推进 → 完成（成果/复盘沉淀）
1. 用户说某任务做完了。
2. agent 先替用户拟一句 `outcome`，询问是否修改、是否记录 `reflection`（复盘可选，不强制）。
3. agent 基于成果+复盘+任务上下文，**即时生成成就陈述 `cv`**：
   - 素材充分 → `cvStatus: ready`，提示"✨ 已沉淀进成就库"。
   - 素材不足以撑起一句有分量的成就 → `cvStatus: pending`（挂起），agent 追问可补充的影响/量化信息。挂起项不会丢，在后续相关对话或用户问起时提醒补全；补全后转 `ready`。
4. 任务完成即从 `tasks.json` 移除，并向 `achievements.jsonl` 追加一条记录。**任务完成一定入库；CV 那条的 ready/pending 与之独立，不阻塞完成。**

### 4.3 问局势 → agent 主动建议
用户问"我现在该干啥"，agent 读取数据，按优先级 + 截止日计算并给出建议：最该动的、可暂时搁置的（如被阻塞的）、有空可推进的。

## 5. 看板（只读，C 布局：顶部聚焦 + 下方按项目）

由 `dashboard.html` 渲染，每 2 秒轮询数据文件自动刷新。

- **顶部"今日聚焦"**：跨项目，由 agent 按优先级 + 截止日算出最该动的任务，**上限 5 条**防止信息过载；每条显示优先级、截止日、下一步动作、是否阻塞。
- **下方按项目分块**：每个未归档项目一块，列出其任务；每条任务显示状态图标（○ 未开始 / ◐ 进行中 / ● 已完成）、优先级、关键信息（截止日 / 下一步 / 阻塞）。
- **草稿高亮**：`draft: true` 的任务以黄色虚线边框 + "草稿待确认"标记呈现。
- **"今天已完成 ✅"可折叠小区**：读取 `achievements.jsonl` 中 `date` 为当天的记录显示，给予成就感，也便于临时反悔（反悔即从成就库移除该条、任务回到 `tasks.json`）。
- **累计成就计数**：显示如"本月已沉淀 N 条成就 · M 条待补充"，呼应长期成就感，并提醒补全 pending 项。

信息密度以当前草样为准（经用户确认不嫌挤）。

## 6. 成果 / CV 总结能力

成就库的设计使 CV 从"事后批量回忆"变为"完成即沉淀"。两种使用方式：

1. **即时沉淀**（见 4.2）：每完成一个任务即生成一条成就陈述入库。
2. **按需重组**：用户给定范围与用途，agent 从 `achievements.jsonl` 筛选 `ready` 条目并按用途模板重组输出 markdown：
   - **述职/答辩**：成果导向，STAR 式，强调影响与推动力。
   - **周报/日报**：平铺直叙，按项目分组，简洁。
   - **复盘/成长**：调取 `reflection`，讲教训与改进。

**底线约束**：agent 只使用 `achievements.jsonl` 中真实记录的内容，**不得编造未发生的事**。素材不足时提示用户补充，绝不虚构。输出为 markdown 供用户取用，默认不入库、不污染数据；用户需要时可存档至 `cv-exports/`。

## 7. 打包形态与服务生命周期

### 7.1 形态
一个 **Skill**，自带资源文件：
```
task-cockpit/                 # skill
├── SKILL.md                  # 工作流说明（拆任务/完成/生成CV 的规则与话术）
├── server.py                 # 常驻本地服务：托管看板 + 数据
└── dashboard.html            # 看板模板（每 2s 轮询自动刷新）

~/.task-cockpit/              # 数据（与 skill 分离）
├── projects.json
├── tasks.json
├── achievements.jsonl
└── cv-exports/
```

### 7.2 服务生命周期
- 用户首次与 skill 交互时，检查服务是否在运行，未运行则启动并告知 URL。
- 服务**不设闲置超时**，用户工作期间持续运行。
- 使用**固定端口**；若服务被关闭（如重启电脑），下次交互时自动重新拉起，URL 不变，浏览器标签无需更换。
- 仅监听 `127.0.0.1`（本机回环），不对外暴露；单人本地使用，无需登录鉴权。

### 7.3 触发方式
用户用自然语言触发，如"我有几个新任务""xx 做完了""我现在该干啥""总结下这周成果"。`SKILL.md` 负责让 agent 识别这些意图并走对应流程。

### 7.4 分发演进路径
- 当前：个人本地 skill。
- 团队分发：将 skill 套入 plugin，自建 git marketplace（如托管于 GitHub），同事 `/plugin marketplace add <仓库地址>` 后安装。每人各自本地实例、数据独立。
- 更广分发：争取进入社区或 Anthropic 官方 marketplace（官方为筛选收录，非提交即上架）。
- **分发前提（现在即守住）**：skill 代码中不得写死任何个人绝对路径，数据目录一律由当前用户家目录推算。

## 8. 风险与边界

- **服务进程依赖**：看板的实时刷新依赖常驻本地服务。已通过"自动拉起 + 固定端口 + 不闲置退出"缓解。
- **CV 真实性**：成就陈述必须源自真实记录，编造会在述职场合造成严重后果。已列为底线约束。
- **团队共享数据**：当前不支持，需共享后端，属未来另开议题。
- **数据备份**：纯本地文件，用户应自行纳入备份（如随家目录备份）。当前不内建备份机制。
